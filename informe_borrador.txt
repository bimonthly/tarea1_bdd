2. ESQUEMA GENERADO POR EL LLM
--------------------------------

El LLM produjo el siguiente esquema relacional:

Torneos (id PK, nombre, videojuego, fecha_inicio, fecha_fin, prize_pool_usd, max_equipos)

Equipos (id PK, nombre, fecha_creacion, capitan_id FK -> Jugadores(id))

Jugadores (id PK, gamertag, nombre_real, email, fecha_nacimiento, pais, equipo_id FK -> Equipos(id))

Sponsors (id PK, nombre, industria)

Sponsors_Torneos (sponsor_id FK -> Sponsors(id), torneo_id FK -> Torneos(id), monto_usd)
  PK compuesta: (sponsor_id, torneo_id)

Inscripciones (id PK, equipo_id FK -> Equipos(id), torneo_id FK -> Torneos(id), fecha_inscripcion)
  UNIQUE: (equipo_id, torneo_id)

Partidas (id PK, torneo_id FK -> Torneos(id), equipo_a_id FK -> Equipos(id),
          equipo_b_id FK -> Equipos(id), fecha_hora, puntaje_a, puntaje_b, fase)
  CHECK: equipo_a_id <> equipo_b_id
  CHECK: fase IN ('fase_grupos', 'cuartos', 'semifinal', 'final')

Estadisticas_Individuales (id PK, partida_id FK -> Partidas(id),
                           jugador_id FK -> Jugadores(id), kos, restarts, assists)
  UNIQUE: (partida_id, jugador_id)


3. COMPARACIÓN CON EL ESQUEMA INDIVIDUAL
------------------------------------------

3.1 Aspectos correctamente capturados por el LLM

El esquema del LLM capturó correctamente las siguientes restricciones
del enunciado:

- La relación de pertenencia fija entre jugadores y equipos, modelada mediante
  equipo_id en la tabla Jugadores.
- El capitán como atributo de Equipos referenciando a Jugadores, con un trigger
  que verifica que el capitán sea efectivamente miembro del equipo.
- El número máximo de equipos por torneo, con un trigger (trg_cupo_torneo) que
  rechaza inscripciones cuando el cupo está lleno.
- Las cuatro fases posibles de una partida: fase_grupos, cuartos, semifinal y
  final, implementadas como restricción CHECK.
- La restricción de que una partida enfrenta exactamente dos equipos distintos,
  implementada con CHECK (equipo_a_id <> equipo_b_id).


3.2 Decisiones de diseño discutibles

Uso de claves primarias surrogadas (SERIAL): El LLM optó por usar
identificadores numéricos autoincrementales en todas las tablas en lugar de
claves naturales. En el esquema individual se utilizaron atributos con
significado semántico como PK (por ejemplo, gamertag para Jugadores o
nombre_equipo para Equipos). Si bien las claves surrogadas son una práctica
estándar en la industria por su eficiencia y estabilidad, las claves naturales
son igualmente válidas cuando se garantiza su unicidad, como se discute en
clases.

Separación de fecha y hora en Partidas: El esquema individual separó fecha y
hora como atributos distintos de tipo int. El LLM los unificó en un único
atributo fecha_hora de tipo TIMESTAMP, lo cual es más correcto desde el punto
de vista del modelo relacional, ya que TIMESTAMP garantiza que el valor
almacenado sea una fecha y hora válida, a diferencia de un entero que no impone
ninguna restricción de dominio.


3.3 Errores identificados

Error principal — modelado incorrecto de Sponsors (esquema individual):

El error más significativo fue detectado en el esquema individual: se modeló
la relación entre Sponsors y Torneos colocando torneo_auspiciado y aporte
directamente dentro de la tabla Sponsors. Esto implica que cada sponsor solo
puede auspiciar un único torneo, violando explícitamente el requerimiento del
enunciado que establece que "un sponsor puede auspiciar múltiples torneos".
Esta es una violación de la forma normal: el monto aportado es un atributo de
la relación entre sponsor y torneo, no del sponsor en sí.

El LLM modeló correctamente esta relación mediante una tabla intermedia
Sponsors_Torneos(sponsor_id, torneo_id, monto_usd), que refleja adecuadamente
la cardinalidad N:M entre sponsors y torneos con el atributo de la relación
correspondiente.

Error de compatibilidad de versiones (error del LLM):

Durante la implementación de la Parte C, el LLM generó código incompatible con
Starlette 1.0.0 (versión instalada automáticamente por pip). Específicamente,
la función TemplateResponse fue invocada con la firma de versiones anteriores,
pasando el objeto request dentro del diccionario de contexto. Esto produjo un
TypeError: unhashable type: 'dict' en tiempo de ejecución. La corrección fue
realizada manualmente por el equipo.


4. CONCLUSIÓN
--------------

El esquema generado por el LLM es relacionalmente correcto y captura
adecuadamente las restricciones del dominio. Sus principales fortalezas son el
uso de claves surrogadas, tipos de datos apropiados, y el modelado correcto de
relaciones N:M. El error más relevante detectado en el proceso correspondió al
esquema individual (modelado incorrecto de Sponsors), mientras que el LLM
cometió un error de compatibilidad de versiones en la implementación, no en el
diseño del esquema. Ambos errores fueron identificados y corregidos por el
equipo durante el desarrollo.
