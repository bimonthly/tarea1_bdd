2. ESQUEMA GENERADO POR EL LLM (post-correcciones hechas por LLM) 
--------------------------------

El LLM produjo el siguiente esquema relacional:

Torneos (id PK, nombre, videojuego, fecha_inicio, fecha_fin, prize_pool_usd, max_equipos)
  CHECK: prize_pool_usd >= 0
  CHECK: max_equipos > 0
  CHECK: fecha_fin >= fecha_inicio

Equipos (id PK, nombre UNIQUE, fecha_creacion, capitan_id FK -> Jugadores(id))

Jugadores (id PK, gamertag UNIQUE, nombre_real, email UNIQUE, fecha_nacimiento, pais,
           equipo_id FK -> Equipos(id) NOT NULL)

Sponsors (id PK, nombre UNIQUE, industria)

Sponsors_Torneos (sponsor_id FK -> Sponsors(id), torneo_id FK -> Torneos(id),
                  monto_usd NOT NULL)
  PK compuesta: (sponsor_id, torneo_id)
  CHECK: monto_usd >= 0

Inscripciones (id PK, equipo_id FK -> Equipos(id), torneo_id FK -> Torneos(id),
               fecha_inscripcion DATE NOT NULL DEFAULT CURRENT_DATE)
  UNIQUE: (equipo_id, torneo_id)

Partidas (id PK, torneo_id FK -> Torneos(id), equipo_a_id FK -> Equipos(id),
          equipo_b_id FK -> Equipos(id), fecha_hora TIMESTAMP NOT NULL,
          puntaje_a INTEGER, puntaje_b INTEGER, fase VARCHAR NOT NULL)
  CHECK: equipo_a_id <> equipo_b_id
  CHECK: fase IN ('fase_grupos', 'cuartos', 'semifinal', 'final')
  CHECK: puntaje_a >= 0
  CHECK: puntaje_b >= 0
  NOTA: puntaje_a y puntaje_b son nullable (permiten partidas no jugadas aún)

Estadisticas_Individuales (id PK, partida_id FK -> Partidas(id),
                           jugador_id FK -> Jugadores(id),
                           kos INTEGER NOT NULL DEFAULT 0,
                           restarts INTEGER NOT NULL DEFAULT 0,
                           assists INTEGER NOT NULL DEFAULT 0)
  UNIQUE: (partida_id, jugador_id)
  CHECK: kos >= 0
  CHECK: restarts >= 0
  CHECK: assists >= 0


3. COMPARACIÓN CON EL ESQUEMA INDIVIDUAL
------------------------------------------

3.1 Aspectos correctamente capturados por el LLM

El esquema del LLM capturó correctamente las siguientes restricciones
del enunciado:

- La relación de pertenencia fija entre jugadores y equipos, modelada mediante
  equipo_id en la tabla Jugadores.
- El capitán como atributo de Equipos referenciando a Jugadores, con un trigger
  que verifica que el capitán sea efectivamente miembro del equipo.
- El número máximo de equipos por torneo, con un trigger (trg_cupo_torneo) que
  rechaza inscripciones cuando el cupo está lleno.
- Las cuatro fases posibles de una partida: fase_grupos, cuartos, semifinal y
  final, implementadas como restricción CHECK.
- La restricción de que una partida enfrenta exactamente dos equipos distintos,
  implementada con CHECK (equipo_a_id <> equipo_b_id).


3.2 Decisiones de diseño discutibles LLM

Uso de claves primarias surrogadas (SERIAL): El LLM optó por usar
identificadores numéricos autoincrementales en todas las tablas en lugar de
claves naturales. En el esquema individual se utilizaron atributos con
significado semántico como PK (por ejemplo, gamertag para Jugadores o
nombre_equipo para Equipos). Si bien las claves surrogadas son una práctica
estándar en la industria por su eficiencia y estabilidad, las claves naturales
son igualmente válidas cuando se garantiza su unicidad, algo que sí ocurre
según las especificaciones del enunciado.

Separación de fecha y hora en Partidas: El esquema individual separó fecha y
hora como atributos distintos de tipo int. El LLM los unificó en un único
atributo fecha_hora de tipo TIMESTAMP, lo cual es más correcto desde el punto
de vista del modelo relacional, ya que TIMESTAMP garantiza que el valor
almacenado sea una fecha y hora válida, a diferencia de un entero que no impone
ninguna restricción de dominio.


3.3 Errores identificados

Esquemas Individuales

Nuestros esquemas individuales presentan varios errores de consistencia, errores lógicos en las relaciones y restricciones de tablas de acuerdo a las especificaciones:
- Llaves foráneas a llaves primarias que no existen (tienen otro nombre).
- Error de modelamiento de partidas, en donde se relaciona a los jugadores con los torneos, y no a través de tablas intermedias.
- Complejidad innecesaria al añadir herencia (isA) para la implementación de capitanes por equipo.
- Ausencia de restricciones de diseño para la tabla de torneos (max equipos, etc).
- Modelado de Sponsors tal que estos auspician un solo torneo cada uno, lo que es incorrecto según la especificación de cardinalidad del enunciado.


Esquema LLM

Error de modelamiento del número mínimo de equipos por torneo:

Si bien el esquema generado por el LLM implementó correctamente la lógica de
equipos máximos por torneo, también permitió el inicio de torneos que no habían
llenado aún el cupo de 8 equipos inscritos, lo que está incorrecto según los
requerimientos del cliente, ya que los torneos deben iniciar con 2 grupos de 4
equipos.

Error de visualización de los datos en la página web (ambigüedad
del enunciado):

Las especificaciones del enunciado dan a entender que la tabla de posiciones
de cada torneo debe mostrar los equipos con partidas realizadas, ya que se
deben mostrar las estadísticas respectivas para cada equipo. El LLM lo implementó
de esta manera. Sin embargo, lógicamente, un equipo también es parte de una tabla
de posiciones a pesar de que tenga 0 partidas jugadas, por lo que se realizó el ajuste.


4. CONCLUSIÓN
--------------

En general, el esquema generado por el LLM es relacionalmente correcto y captura
adecuadamente las restricciones del enunciado. Esto contrasta con el modelado de los
esquemas individuales, donde hubieron errores en la creación de llaves y en las referencias a 
llaves dentro de las relaciones, junto a una complejidad innecesaria que aumenta la posibilidad 
de errores e inconsistencias. Las principales
fortalezas del esquema del LLM son el
uso de claves surrogadas, tipos de datos apropiados, y el modelado correcto de
relaciones N:M. Cometió un error de modelado respecto a los equipos mínimos por torno, 
además de algunos bugs en la página web. Los errores
fueron identificados y corregidos por el
equipo durante el desarrollo.
